## 決定事項

- 将棋ロジック（盤面・合法手・終局判定・AI）は python-shogi 等の既存将棋ライブラリを使わず自作する（2026-07-04 決定）

## Phase分割案

依存関係の順に並べている: データモデル → 移動生成 → 合法手判定 → 対局進行 → AI → GUI。
自作方針のため、旧「合法手生成」を「駒の移動生成（疑似合法手）」と「合法手判定（反則の除外）」の2段に分けている。
AIをGUIより先に置くのは、CLI対局さえあればAIの動作検証ができ、GUIはどのPhaseにも依存されないため。

| Phase | 内容 | 完了条件 | 状態 |
|---|---|---|---|
| SHOGI-0 | 設計。CLAUDE.md / Skill / README / 本計画の骨子作成 | 骨子案が揃い人間レビュー待ちであること | 完了 |
| SHOGI-1 | 盤面・駒のデータモデル + SFEN入出力 | 平手初期局面と任意局面をSFENで読み書きでき、ラウンドトリップのテストが green | 完了 |
| SHOGI-2 | 駒の移動生成 | 任意局面の疑似合法手（盤上の駒の移動先・成り/不成の候補・駒打ちの候補）を列挙できる。盤外と味方駒のあるマスは除外する。王手・反則の判定はこのPhaseではしない（合法手判定は SHOGI-3） | **次フェーズ** |
| SHOGI-3 | 合法手判定 | 疑似合法手から反則手を除外し、真の合法手を確定できる。王手検出、王手放置の禁止、二歩、行き所のない駒、打ち歩詰めに対応する | 未着手 |
| SHOGI-4 | 対局進行（CLI・人対人） | CLIで平手の対局を最初から最後まで指せる。詰み・投了で終局判定できる（千日手等の扱いは着手前に判断（要確認）） | 未着手 |
| SHOGI-5 | AI v1 + 人対AI | 合法手から手を選ぶAI（ランダム or 1手読み）とCLIで対局できる | 未着手 |
| SHOGI-6 | AI v2（評価関数 + 探索） | 駒割ベースの評価関数とminimax/αβ探索で、AI v1 に明確に勝ち越すAIになる | 未着手 |
| SHOGI-7 | GUI | 技術選定（pygame / Tkinter / ブラウザ）の上、盤面表示とマウス/キー操作で対局できる | 未着手 |

- MVPの区切り: SHOGI-5 完了（=AIと将棋が指せる）を最初のMVPとする案（要確認）
- 各Phaseの実装は `.claude/skills/phase-implementation` に従って行う

### SHOGI-1 の実施内訳（完了済み）

実際にはレビュー可能な単位で以下のサブフェーズに分けて実施した:

1. 環境構築（1a）: `src/` + `tests/` 構成、pyproject.toml、pytest 動作確認
2. Piece（1b）: `piece.py` — Color / PieceType（成駒6種を含む14種）/ Piece
3. Board（1c）: `board.py` — 9x9 の盤面、`Piece | None` 保持、座標系の確定
4. 初期局面（1d）: `initial_position.py` — 平手40枚の配置
5. SFEN出力（1e-1）: `sfen.py` — 盤面部の書き出し（空きマス圧縮）
6. SFEN読込（1e-2）: 盤面部の読み込みとバリデーション（段数・マス数・連続数字・不正文字は ValueError）

手番・持ち駒・手数を含む完全な SFEN と Position 概念は未実装（対局進行系の Phase で導入する）。

## テスト方針（pytest・たたき台）

- ユニットテスト中心。ゲームロジックをGUI/入出力から分離し、ロジック層のみをpytestの対象とする（GUIは手動確認）
- 着手順（Phaseに対応）:
  1. SFEN入出力のラウンドトリップ（parse → dump が一致）
  2. 駒の移動生成: 駒種ごとの動きの突き合わせ。盤上に対象駒だけを置いた局面で移動先一覧を照合する（全駒種・成駒を網羅）。端や敵陣境界（成り候補の有無）を境界値にする
  3. 合法手判定: 既知局面の合法手リスト/手数との突き合わせ。平手初期局面の合法手30手など、答えが確定している局面をフィクスチャにする。二歩・打ち歩詰め・王手放置は反則になる局面/ならない局面をペアでテストする。余力があればperft（数手先の局面数の照合）を導入（要確認）
  4. 終局判定: 詰み局面・詰みでない王手局面の判定
  5. AI: 「常に合法手を返す」「詰みがあるとき詰み手を選ぶ（v2）」などの性質テスト。強さそのものはテストしない
- 局面フィクスチャはSFEN文字列で管理し、盤面図コメントを添える
- 各Phaseの完了条件に「テスト全件 green」を含める

## Skill構成

| Skill | 用途 | 状態 |
|---|---|---|
| phase-implementation | Phaseを1つ実装する際の手順 | 骨子作成済み |
| bug-fix | 不具合修正（再現テスト→最小修正） | 骨子作成済み |

追加候補（必要になった時点で作成し、先取りしない）:
- 局面デバッグ支援（SFENの盤面可視化・合法手ダンプ）— SHOGI-2 以降で必要性を判断
- 引継ぎ文作成 — 運用してみて手順が固まったら

## リスク洗い出し（初期案）

| リスク | 影響 | 軽減策 |
|---|---|---|
| 移動生成・合法手判定の網羅漏れ（特に打ち歩詰め・成り条件・打ち駒の制約） | ゲームとして破綻 | 駒種ごとの動きの網羅テスト、既知局面との突き合わせテスト、perft検証（要確認） |
| 千日手・持将棋・入玉宣言の判定が複雑 | Phase 4 の肥大化 | 対応範囲を Phase 4 着手前に明示的に決め、対象外は「対応しない」と明記 |
| Python の速度で AI 探索が遅い | Phase 6 で体感品質低下 | 探索深さを config 化して制限。最適化は問題が出てから |
| GUI 技術未定 | Phase 7 の見積り不能 | Phase 7 まで決定を遅延（他Phaseに影響しない構成にしてある） |
| スコープ膨張（先取り実装） | MVP遅延 | CLAUDE.md 開発方針 + phase-implementation スキルで抑制 |
