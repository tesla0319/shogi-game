SHOGI-0 で作成した初期案。人間のレビュー・承認を経て確定する。

## 決定事項

- 将棋ロジック（盤面・合法手・終局判定・AI）は python-shogi 等の既存将棋ライブラリを使わず自作する（2026-07-04 決定）

## Phase分割案

依存関係の順に並べている: データモデル → 移動生成 → 合法手判定 → 対局進行 → AI → GUI。
自作方針のため、旧「合法手生成」を「駒の移動生成（疑似合法手）」と「合法手判定（反則の除外）」の2段に分けている。
AIをGUIより先に置くのは、CLI対局さえあればAIの動作検証ができ、GUIはどのPhaseにも依存されないため。

### SHOGI-1 の実施内訳（完了済み）

実際にはレビュー可能な単位で以下のサブフェーズに分けて実施した:

1. 環境構築（1a）: `src/` + `tests/` 構成、pyproject.toml、pytest 動作確認
2. Piece（1b）: `piece.py` — Color / PieceType（成駒6種を含む14種）/ Piece
3. Board（1c）: `board.py` — 9x9 の盤面、`Piece | None` 保持、座標系の確定
4. 初期局面（1d）: `initial_position.py` — 平手40枚の配置
5. SFEN出力（1e-1）: `sfen.py` — 盤面部の書き出し（空きマス圧縮）
6. SFEN読込（1e-2）: 盤面部の読み込みとバリデーション（段数・マス数・連続数字・不正文字は ValueError）

手番・持ち駒・手数を含む完全な SFEN と Position 概念は未実装（対局進行系の Phase で導入する）。

### SHOGI-3 の実施内訳（サブフェーズ）

SHOGI-3 は「盤上移動の合法手判定」に責務を絞り、レビュー可能な単位で以下のサブフェーズに分ける。
持ち駒・駒打ちと、それに依存する反則（二歩・打ち歩詰め・打ちの行き所のない駒）は SHOGI-4 冒頭へ移動した。
手番・持ち駒を持つ Position 概念は導入せず、手番は関数の引数（color）で受け取る。

1. 盤面シミュレート基盤（3a・完了）: `board.py` に `Board.copy()`、`rules.py`（新規）に `board_after_move`（取り＝盤から除去のみ・成り＝駒種変換）。持ち駒には加えない
2. 王手検出（3b）: `rules.py` に `find_king` / `is_attacked` / `is_in_check`。既存の `generate_piece_moves`（movegen）を利き判定に流用する
3. 合法手生成（3c）: `rules.py` に `generate_legal_moves(board, color)`。疑似合法手を「指した後に自玉が王手なら除外」の単一フィルタで絞り、王手放置・自殺手・ピンをまとめて満たす

### SHOGI-4 の実施内訳（駒打ち基盤 4a〜4h・USI指し手変換 4i・局面表示 4j・対局状態管理 4k・CLI対局ループ 4l・自動終局判定 4m 完了）

SHOGI-4 冒頭の「駒打ち基盤」を、レビュー可能な単位で以下の 4a〜4h に分けて実施した。
これで持ち駒・駒打ちを含む合法手生成と着手後の状態反映が揃った。続いて対局進行本体の
下ごしらえとして、USI 指し手表記と Move の相互変換（4i）、局面のテキスト表示（4j）、
盤面・両者持ち駒・手番を集約する対局状態管理（4k）を実装し、これらを組み合わせた
人対人の CLI 対局ループ（4l）、合法手0件による自動終局判定（4m）まで実装した。
これで SHOGI-4（人対人 CLI 対局）は完了。

1. Hand（4a・完了）: `hand.py` — 持ち駒（基本7駒種の枚数）を独立管理。玉・成駒は保持しない
2. Move の駒打ち表現（4b・完了）: `move.py` に `drop_piece_type` / `Move.drop()` / `is_drop`。盤上移動 Move は不変
3. 駒打ちの盤面反映（4c・完了）: `rules.py` に `position_after_move(board, hand, color, move)`。駒打ちを盤と持ち駒へ反映（合法性は検証しない）
4. 駒打ち候補生成（4d・完了）: `movegen.py` に `generate_drop_moves(board, hand, color)`。空きマスへの `Move.drop` を列挙
5. 二歩・行き所のない駒の除外（4e・完了）: `generate_drop_moves` 側で、同筋の自分の未成歩（二歩）と最奥段（行き所のない駒）への打ちを除外。行き所判定は盤上移動の `_must_promote` を流用
6. 駒打ちの合法手統合（4f・完了）: `generate_legal_moves(board, color, hand=None)`。hand を渡すと駒打ちも「指した後に自玉が王手なら除外」でフィルタ。`hand=None` は従来どおり盤上移動のみ（後方互換）
7. 打ち歩詰めの除外（4g・完了）: `generate_legal_moves` に `check_uchifuzume`（内部再帰制御）。歩打ちで相手玉が詰む手を反則として除外
8. 駒取り時の持ち駒加算（4h・完了）: `position_after_move` の盤上移動分岐で、取った相手駒を持ち駒に加える（成駒は素の駒に戻す・玉は加えない）。`board_after_move` は盤面専用のまま不変
9. USI 指し手変換（4i・完了）: `usi.py`（新規）に `move_from_usi` / `move_to_usi`。通常手・成り手・駒打ちを USI 表記と Move で相互変換する。形式変換のみで盤面上の合法性は判定しない（`sfen.py` が Board と SFEN を相互変換するのと同じ役割分担）。不正な表記は ValueError
10. 局面表示（4j・完了）: `display.py`（新規）に `position_to_text(board, black_hand, white_hand, side_to_move)`。盤面・両者持ち駒・手番を人間可読な1つの文字列にする純関数（print はしない・ANSI装飾なし）。駒文字は SFEN に合わせ、盤の向きは先手視点固定。座標ラベルは USI（筋 1〜9・段 a〜i）に一致
11. 対局状態管理（4k・完了）: `position.py`（新規）に `Position`（frozen dataclass。盤面・先手持ち駒・後手持ち駒・手番を保持）と `Position.apply_move(move)`（既存の `position_after_move` へ委譲し、相手持ち駒を引き継いで手番を反転。元の Position は壊さない）、`create_hirate_position()`。責務は状態保持と着手適用に限り、合法性・終局・履歴は扱わない
12. CLI対局ループ（4l・完了）: `cli.py`（新規）に `run_game(input_fn, output_fn)`。平手初期局面から手番側に USI 合法手または `resign` を入力させ、局面表示→入力→着手適用を反復する。入出力は注入可能（既定は input / print）。非合法手・USI形式不正・入力終了（EOF）を処理する。`__main__.py`（新規）で `python -m shogi` から起動する。自動終局判定（詰み）は含まない（SHOGI-4m）
13. 自動終局判定（4m・完了）: `cli.py` の対局ループに追加。各手番で入力を求める前に手番側の合法手を生成し、0件なら終局する。`is_in_check` が真なら詰み（相手の勝ち）、偽なら「合法手なし」で手番側の負け。判定は既存の `generate_legal_moves` と `is_in_check` の組み合わせだけで行い、終局専用モジュール・列挙型は導入しない。勝敗は表示するのみで `Position` には保持しない。テスト用に `run_game(start_position=...)` で任意局面から開始できる（既定は平手）

これで SHOGI-4a〜4m までの pytest 全601件が green。

制限事項（今後の開発の制約として明記。前倒し実装はしない）:

- Position は導入済み（4k）だが、責務は状態保持と着手適用に限る。合法性判定・終局判定・指し手履歴は持たず、呼び出し側（対局ループ）が扱う
- Board は盤面専用（持ち駒を持たせない）。持ち駒は Hand で独立管理する
- 両者持ち駒の保持・手番進行は `Position`（4k）が担う。着手反映は `Position.apply_move(move)` を使う（内部で `position_after_move(board, hand, color, move)` に委譲する）
- `generate_legal_moves` は合法手（Move）の一覧を返すのみで、着手後の Hand は返さない
- 打ち歩詰め判定は MVP 実装。相手の回避手は盤上移動のみを見ており、相手の持ち駒による合駒は未対応（過剰に「打ち歩詰め」と判定して除外し得る安全側の割り切り）

## テスト方針（pytest・たたき台）

- ユニットテスト中心。ゲームロジックをGUI/入出力から分離し、ロジック層のみをpytestの対象とする（GUIは手動確認）
- 着手順（Phaseに対応）:
  1. SFEN入出力のラウンドトリップ（parse → dump が一致）
  2. 駒の移動生成: 駒種ごとの動きの突き合わせ。盤上に対象駒だけを置いた局面で移動先一覧を照合する（全駒種・成駒を網羅）。端や敵陣境界（成り候補の有無）を境界値にする
  3. 合法手判定: 既知局面の合法手リスト/手数との突き合わせ。平手初期局面の合法手30手など、答えが確定している局面をフィクスチャにする。二歩・打ち歩詰め・王手放置は反則になる局面/ならない局面をペアでテストする。余力があればperft（数手先の局面数の照合）を導入（要確認）
  4. 終局判定: 詰み局面・詰みでない王手局面の判定
  5. AI: 「常に合法手を返す」「詰みがあるとき詰み手を選ぶ（v2）」などの性質テスト。強さそのものはテストしない
- 局面フィクスチャはSFEN文字列で管理し、盤面図コメントを添える
- 各Phaseの完了条件に「テスト全件 green」を含める
