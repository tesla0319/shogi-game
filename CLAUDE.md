- 言語: Python 3.x（要確認: バージョン固定するか。3.12 を想定）

## 将棋ドメインの取り決め【固有】

<!-- 局面・指し手の表記ゆれを防ぐための取り決め。テストの期待値もこれに従う -->

- 局面の外部表記は SFEN を正とする（入出力・テストのフィクスチャは SFEN 文字列で書く）
- 指し手の外部表記は USI 形式を正とする（例: `7g7f`、駒打ち `P*5e`、成り `8h2b+`）
- 先手/後手は USI に合わせて先手=b(black)、後手=w(white) とする
- 盤面の内部表現（SHOGI-1c で決定）: Board は 9x9 のリストのリストで各マスに Piece または None を保持する。座標は file（筋）=1〜9（先手から見て右端が1）、rank（段）=1〜9（先手から見て奥が1。USI の段 a〜i に 1〜9 がこの順で対応）。盤外座標は ValueError
- ルールの正は日本将棋連盟の対局規定とする。特殊ルールの対応範囲は次のとおり:
  - Phase 2（駒の移動生成）で対応: 各駒種の動き、成り/不成の候補生成
  - Phase 3（合法手判定）で対応: 二歩、行き所のない駒、打ち歩詰め、王手放置の禁止
  - SHOGI-4 では対象外: 千日手、連続王手の千日手、持将棋、入玉宣言

## 開発方針【共通】

- MVP優先。Phase単位（レビュー可能な最小の変更単位）で実装する
- 一気に全機能を実装しない。将来のPhaseの分を先取りしない
- 変更範囲を最小化する。依頼されていないファイル・関数に触らない
- 既存仕様を勝手に変更しない。挙動が変わる修正は実装前に必ず確認を取る
- 不明点は推測せず、着手前に質問する

## コード方針【共通】

- 初学者でも理解しやすいコードを書く。可読性 > 短さ・巧妙さ
- 複雑な最適化・過剰な抽象化を避ける
- 1関数=1責務。関数やクラスの責務を明確にする
- 「なぜその実装にしたか」をコメントで残す（「何をしているか」はコードで読めるので書かない）
- config化できる値（上限値・タイムアウト・パス・閾値）は定数化を提案する

## 実装の型【育成】

<!-- Phase を実装しながらここに集約する -->
- ゲームロジック（局面・合法手・終局判定）は入出力・GUI から分離し、純粋な関数/クラスとして書く（テスト可能性のため）
- 指し手の外部表記（USI）と Move の相互変換は `usi.py` に集約する（Board と SFEN を相互変換する `sfen.py` と対をなす役割分担）。USI 変換は表記の形だけを扱い、盤面上の合法性は判定しない
- 局面の表示は `display.py` の純関数（`position_to_text`）に集約する。print はせず文字列を返すだけ（表示先は呼び出し側が決める）。駒文字は SFEN に合わせ、盤の向きは先手視点固定
- 対局状態は `position.py` の `Position`（frozen dataclass）へ集約する。保持するのは盤面・両者の持ち駒・手番だけ。着手適用は `Position.apply_move()` を使い、元の局面を壊さず次局面を返す。Position では合法性・終局・履歴を判定・保持しない（呼び出し側の責務）
- CLI 対局ループは `cli.py` の `run_game(input_fn, output_fn)` へ集約する。入力・出力は注入可能にして対局ロジックを標準入出力から分離する（テスト可能性のため）。CLI 起動方法は `python -m shogi`（`__main__.py`）
- 終局判定は入力受付の前に行う（手番側の合法手が0件かで判定）。合法手生成・王手判定は `rules.py` の既存関数（`generate_legal_moves` / `is_in_check`）へ委譲し、CLI 層は新しい判定ロジックを持たない。勝敗は表示するのみで `Position` へは保持しない

## テスト方針【共通】

- 追加・変更した関数には 正常系 + 境界値 + 異常系 のテストを付ける
- 修正前に既存テスト全件 green を確認し、修正後も全件 green を維持する
- テストを消したり skip にして green にしない
- 完了報告には「想定される不具合・境界値・残っているリスク」を必ず含める
