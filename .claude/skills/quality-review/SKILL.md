# quality-review スキル

## 目的

設計案または実装差分に対して、テストエンジニア視点のレビューを固定観点で行い、
Must/Should/Could に分類された指摘一覧を返す。修正は行わない（指摘と提案のみ）。

## 原則

- CLAUDE.md の共通ルール（開発方針・コード方針・テスト方針・やらないこと）に従う
- 指摘には必ず「理由（何が困るか）」を付ける
- 指摘の重要度は3段階で分類する:
  - **Must**: 不具合・仕様違反・データ破壊など放置できないもの。このPhase内で必ず対応
  - **Should**: 高確率で将来の不具合になるもの。このPhase内での対応を推奨
  - **Could**: 無くても動く改善提案。別Phase候補として記録のみ。今は対応しない
- 「綺麗になる」だけの指摘は Could にも入れない（出力しない）

## モード判定

このスキルは2つのモードを持つ。

- **設計レビューモード**: コードを書く前の設計案・Phase計画が入力
- **実装レビューモード**: 実装差分（変更ファイル・diff・コード）が入力

設計案か実装差分か不明な場合は、入力内容から推定してレビューを開始する
（コード・diff・ファイルパスが含まれれば実装、文章・表・分割案が中心なら設計と推定する）。
推定できない場合のみユーザーへ確認する。不要な確認で処理を止めないことを優先する。
推定した場合は、レビュー結果の冒頭に「モード: 実装（推定）」のように推定であることを明示する。

## 手順

### 1. 入力の確認
- レビュー対象と、対象Phaseの完了条件を確認する。完了条件が不明なら質問する
- モードを判定する（上記「モード判定」に従う）

### 2. 前提の読み込み
- CLAUDE.md（実装の型・やらないこと）と、関連する既存コードを読む
- 差分だけを見て判断しない。呼び出し元・共有状態も確認する

### 3. 観点別チェック
- 下記チェックリストを上から順に全項目確認する（該当なしも「該当なし」と明示する）

### 4. 重要度分類
- 指摘を Must/Should/Could に分類し、各指摘に「理由（何が困るか）」と「再現条件または具体例」を付ける

### 5. 過剰指摘の自己検閲
- 「綺麗だから」だけの指摘、MVP範囲外への機能追加提案を出力から削除する

### 6. 報告
- 出力フォーマットに従って報告する

## チェックリスト

- [ ] **完了条件との整合**: 今回のPhaseの完了条件を満たしているか。完了条件に無いものを作っていないか（先取り実装）
- [ ] **責務分離**: 1関数=1責務になっているか。UIとロジック、入出力と計算が混ざっていないか
- [ ] **既存設計との整合**: CLAUDE.mdの「実装の型」に従っているか。既存の同種処理と書き方が揃っているか
- [ ] **影響範囲**: 変更が波及する呼び出し元・共有状態・設定ファイルを列挙したか。触っていないはずのファイルが変わっていないか
- [ ] **回帰リスク**: 既存の挙動が変わる箇所はあるか。あるなら意図的か・確認済みか
- [ ] **例外・異常系**: 外部入力・ファイルI/O・subprocess・ネットワークの失敗時の挙動が定義されているか
- [ ] **境界値**: 空・0・最大値・重複・不正型の扱いが定義されているか
- [ ] **テスト充足**: 変更箇所に対応するテストがあるか。不足があれば test-design スキルでの観点出しを提案する（このスキル内でテスト設計まではしない）
- [ ] **過剰設計チェック**: 今回不要な抽象化・将来のための拡張ポイント・使われない引数がないか
- [ ] **セキュリティ基本**: パス操作・コマンド実行・外部入力の埋め込みに危険なパターンがないか

## 出力フォーマット

```markdown
## レビュー結果: <対象> (モード: 設計 / 実装 ※推定の場合は明示)

### 結論
<マージ可 / Must対応後にマージ可 / 設計見直し推奨> を1行で

### 指摘一覧
| # | 重要度 | 観点 | 指摘 | 理由（何が困るか） | 再現条件・具体例 |
|---|---|---|---|---|---|

### 確認済みで問題なかった観点
<チェックリストのうち該当なしだった項目を列挙>

### 別Phase候補（今回は対応しない）
<Could相当の改善案。無ければ「なし」>
```

## 適用例

- 「Phase 3の実装が終わったのでレビューして」（実装レビューモード）
- 「この設計でPhase分割に進んでいいか見て」（設計レビューモード）
- 外部で作った設計案を貼って「実装前にレビューして」

## 適用しないケース

- コードの修正・リファクタの実行そのもの（→ phase-implementation）
- テストケースの具体設計（→ test-design に引き継ぐ）
- 「とりあえず動かして確認して」などの動作確認作業
- コーディング規約の全ファイル一括チェック（大規模リファクタの入口になるため）
