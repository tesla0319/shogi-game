# phase-implementation スキル

## 目的

依頼された1つのPhaseについて、development-plan.md の完了条件をすべて満たし、テスト全件 green の状態で報告できていること。

## 原則

- 1回の作業単位は1Phase。複数Phaseをまとめて実装しない
- development-plan.md に書かれた完了条件が唯一のゴール。書かれていない機能は作らない
- 将棋のルール・表記は CLAUDE.md「将棋ドメインの取り決め」に従う（SFEN/USI）
- 不明点は推測で埋めず、着手前に質問する

## 手順

### 1. 着手前（実装を書き始める前に必ず）

- docs/development-plan.md から該当Phaseの完了条件を読み、1〜2行で言語化して依頼者に確認する
- 触るファイル・追加する関数・追加するテストを列挙する（触らない範囲も明示する）
- 前Phaseまでの成果物と衝突しうる点があれば洗い出し、判断を仰ぐ
- 該当Phaseに（要確認）マークの未決事項が残っていれば、先に判断を仰ぐ

### 2. 実装

- CLAUDE.md の「実装の型」に必ず合わせる
- ゲームロジックは入出力・GUIから分離した純粋な関数/クラスとして書く
- 局面を扱う関数の入出力は SFEN/USI 表記と相互変換できる形にする
- Phase完了時に「実装の型」に追記すべき決まりごとが生まれたら、追記案を報告に含める

### 3. テスト

- 正常系 + 境界値 + 異常系 のテストを付ける
- 既存テスト全件 green を維持する
- 局面のフィクスチャは SFEN 文字列で書く（盤面図のコメントを添える）
- 合法手まわりの実装では、既知局面との突き合わせテスト（development-plan.md「テスト方針」参照）を必ず入れる

### 4. 報告（完了時に必ず含める）

- 変更ファイル一覧（未変更の主要ファイルも明示する）
- 追加・更新したテストとテスト結果（全件数・成否）
- 想定される不具合・境界値・残っているリスク
- development-plan.md の該当Phaseのステータス更新案と、CLAUDE.md【育成】節への追記案

## やらないこと

- 完了条件に含まれない機能の先取り実装（次Phase以降の内容）
- 「ついでに」の周辺リファクタ（別途提案する）
- development-plan.md の完了条件そのものの変更（変更が必要なら実装前に提案する）
