# release-readiness スキル

## 目的

Phase完了・コミット・リリースの前に、出荷準備が整っているかをチェックリストで確認し、
Go / 条件付きGo / No-Go の判定と残作業の一覧を返す。実装・修正・品質指摘は行わない。

## 原則

- CLAUDE.md / AGENT.md の共通ルールに従う
- 本スキルは**準備の充足を確認する**だけで、品質の中身は評価しない（中身は quality-review の領分）
- 各項目は「確認済み / 未確認 / 該当なし」で判定し、未確認項目には担当スキルまたは作業を紐づける
- No-Go の場合も残作業を列挙して返す（判定だけで終わらない）
- テストの実行・結果確認は行ってよい（確認行為であり修正ではない）

## やること / やらないこと

**やること**
- Phase完了条件の充足確認
- テスト実施状況の確認（全件実行と結果の確認）
- レビュー消化の確認（quality-review等のMust指摘が残っていないか）
- ドキュメント更新の要否確認状況のチェック
- 未解決事項・保留事項の列挙
- コミット可否・次Phaseへ進めるかの判定

**やらないこと**
- 実装・修正（不足が見つかっても直さない。残作業として列挙する）
- 品質の中身の評価・コードレビュー（→ quality-review）
- ドキュメント更新内容の作成（→ documentation-update）
- テストの追加・設計（→ test-design）

## 手順

1. **対象の確認**: 何を出荷したいのか（Phase完了 / コミット / リリース）と、対象Phaseの完了条件を確認する
2. **完了条件チェック**: 完了条件が満たされているか、動作確認の記録があるかを確認する
3. **テスト状況の確認**: pytestを全件実行し、結果（全件数・成否）を確認する。追加されたテストが完了条件に対応しているかも見る
4. **レビュー消化の確認**: quality-review 等の指摘のうち Must が未対応で残っていないかを確認する。レビュー未実施なら「未確認」として残作業に入れる
5. **ドキュメント確認**: README / CHANGELOG / docs の更新要否が確認済みかを見る。未確認なら documentation-update の実施を残作業に入れる（要否の判定自体は行わない)
6. **未解決事項の列挙**: 保留した確認事項・既知の残リスク・コミット対象外ファイルの混入がないかを確認する
7. **判定と報告**: Go / 条件付きGo（残作業完了後に可）/ No-Go を判定し、出力フォーマットで報告する

## チェックリスト

- [ ] Phase完了条件が満たされているか（「〜ができる」を確認したか）
- [ ] pytest全件が green か（実行結果を確認したか。未実行のまま判定していないか）
- [ ] 追加・変更した機能に対応するテストが存在するか
- [ ] レビューのMust指摘が未対応で残っていないか
- [ ] README / CHANGELOG / docs の更新要否が確認済みか
- [ ] 未解決の確認事項・保留事項が残っていないか（残っている場合、出荷を妨げるか判定したか）
- [ ] 生成物・ビルド成果物・一時ファイルがコミット対象に混ざっていないか
- [ ] 次Phaseに進む前に合意すべき事項がないか

## 出力フォーマット

```markdown
## 出荷準備判定: <対象Phase / コミット / リリース>

### 判定
<Go / 条件付きGo / No-Go> を1行 + 理由

### チェック結果
| 項目 | 状態(確認済み/未確認/該当なし) | 備考 |
|---|---|---|
| 完了条件 | | |
| テスト実施 | | |
| レビュー消化 | | |
| ドキュメント | | |
| 未解決事項 | | |
| コミット対象 | | |

### 残作業（条件付きGo / No-Go の場合）
| 作業 | 担当スキルまたは対応 |
|---|---|

### 次Phaseへ進めるか
<進める / 残作業完了後に進める / 要相談>
```

## 適用例

- Phase実装完了後の「コミットしていい？」の判定
- 複数Phaseをまとめてリリースする前の最終確認
- 「次のPhaseに進んでいいか」の確認
- 久しぶりに再開するプロジェクトの現状棚卸し（未解決事項の確認）

## 適用しないケース

- コード品質の指摘出し（→ quality-review）
- 見つかった不足の修正作業（→ bug-fix / phase-implementation）
- ドキュメント更新文面の作成（→ documentation-update）
- CI/CDパイプラインの構築・リリース作業の自動化

## 他Skillとの責務境界

- **quality-review**: 品質の中身（コードの良し悪し）を評価する。本スキルはその結果を含む「準備の充足」だけを確認する。quality-review = 検品、本スキル = 出荷前の梱包チェック、という関係
- **documentation-update**: ドキュメント更新の要否判定と文面作成を担う。本スキルは「その確認が済んでいるか」だけを見る
- **phase-implementation / bug-fix**: 残作業の実施先。本スキルは作業を列挙するだけで実施しない
- **phase-design**: 「次Phaseへ進めるか」がGoになった後の、次Phaseの計画を担う
