# documentation-update スキル

## 目的

実装変更を入力として、ドキュメント（README / CHANGELOG / docs / サンプル / コメント / 利用手順）の
更新要否を判定し、更新が必要な場合は「更新対象・更新理由・文面案」を整理して返す。
文面案の作成までを担い、実ファイルの編集は依頼された場合のみ最小限で行う。

## 原則

- CLAUDE.md / AGENT.md の共通ルールに従う
- 更新は必要最小限にする。変更に関係しないドキュメントの手直し・全面書き直しをしない
- 判定基準は「読者の行動が変わるか」: 使い方・導入手順・公開インターフェースが変わったときのみ更新する
- 文面はドキュメントごとの読者に合わせる（README=初見の利用者 / CHANGELOG=変更を追う人 / コメント=コードを読む人）
- 実装と食い違うドキュメントを見つけた場合、修正するのはドキュメント側（実装を変えたくなったら別途相談）

## やること / やらないこと

**やること**
- 実装変更を入力とした、ドキュメント更新の要否判定
- 更新対象の列挙（README / CHANGELOG / docs / サンプル / コメント / 利用手順）
- 更新理由の明記（読者の何が変わるか）
- 文面案の作成（追記・修正の具体的な内容）
- ドキュメント間の整合性確認（説明の食い違いがないか）

**やらないこと**
- コード修正・実装作業（→ phase-implementation / bug-fix）
- 実装レビュー（→ quality-review）
- テスト設計（→ test-design）
- 変更に関係しないドキュメントの改善・再構成（別Phaseとして提案）

## 手順

1. **変更内容の確認**: 実装差分または完了報告を入力として受け取る。何が変わったのかが不明なら確認する
2. **影響判定**: 変更を次の観点で仕分ける
   - 使い方・操作が変わる → README・利用手順・サンプル
   - 導入・セットアップが変わる → README（導入節）・利用手順
   - 公開インターフェース（関数・コマンド・設定項目）が変わる → docs・サンプル
   - 挙動の変更・機能追加・修正 → CHANGELOG
   - 実装の意図・制約が変わる → コード内コメント
3. **要否の判定**: 該当がなければ「更新不要」と理由を返して終了する
4. **文面案の作成**: 更新対象ごとに、追記・修正する具体的な文面を最小限で作る
5. **整合性確認**: 今回の文面が他のドキュメントの記述と食い違わないか確認する
6. **報告**: 出力フォーマットに従って返す。ファイル編集まで依頼された場合は文面案どおりに最小限で編集する

## チェックリスト

- [ ] 更新要否を「読者の行動が変わるか」で判定したか
- [ ] 更新不要の場合、その理由を明示したか
- [ ] 更新対象ごとに理由（読者の何が変わるか）を付けたか
- [ ] 文面案が変更内容と一致しているか（実装と食い違っていないか）
- [ ] CHANGELOGの記載が「利用者に見える変化」で書かれているか（内部実装の詳細になっていないか）
- [ ] サンプル・利用手順が変更後の実装で実際に動く内容か
- [ ] 変更に関係しないドキュメントの手直しを混ぜていないか
- [ ] ドキュメント間で説明が食い違っていないか

## 出力フォーマット

```markdown
## ドキュメント更新整理: <対象変更>

### 判定
<更新必要 / 更新不要> を1行 + 理由

### 更新一覧（更新必要の場合）
| 更新対象 | 更新理由（読者の何が変わるか） | 更新種別(追記/修正/削除) |
|---|---|---|

### 文面案
<更新対象ごとの具体的な追記・修正内容>

### 整合性メモ
<他ドキュメントとの食い違い・実装との食い違い。無ければ「なし」>

### 対象外とした候補
<検討したが今回は更新不要と判断したドキュメントと理由>
```

## 適用例

- Phase完了後の「READMEとCHANGELOG、何を書けばいい？」
- コマンド・設定項目を追加したときの利用手順・サンプルの更新整理
- release-readiness で「ドキュメント未確認」となった項目の解消
- 「ドキュメントと実際の動きが違う」と気づいたときの更新箇所の洗い出し

## 適用しないケース

- ドキュメントサイトの構築・ドキュメント全体の再構成（別Phaseとして提案）
- コード修正を伴う対応（実装と食い違う原因が実装側のバグなら → bug-fix）
- 新規プロジェクトのREADME初回作成（テンプレートからの作成であり、変更に伴う更新ではない）
- コミットメッセージの作成（Git運用の範囲。本スキルはドキュメントファイルのみ）

## 他Skillとの責務境界

- **phase-implementation**: 実装と完了報告を担う。本スキルはその報告を入力として、ドキュメントへの反映内容を整理する（実装の後工程）
- **release-readiness**: 「ドキュメント更新の要否確認が済んでいるか」だけをチェックする。要否の判定と文面作成は本スキルが担う
- **quality-review**: コードの品質を見る。本スキルはドキュメントと実装の整合のみ見る
- **bug-fix**: ドキュメントと実装の食い違いの原因が実装側の不具合だった場合、修正を委譲する
