# dependency-review スキル

## 目的

新しいライブラリ・依存・外部ツールの導入提案を審査し、
採用 / 縮小して採用 / 見送り の判定と、見送り時の代替案を返す。導入作業・実装は行わない。

## 原則

- CLAUDE.md / AGENT.md の共通ルールに従う
- **新規依存は「原則追加しない」から出発する**。必要性が証明されたときのみ採用する
- 必要性の判定は次の2つの質問で行う。どちらもNoなら見送り:
  1. これが無いと今回のPhaseの完了条件を満たせないか
  2. 自作すると、プロジェクトの本質でない大きな実装（車輪の再発明）が必要になるか
- 見送り判定のときは、必ず代替案（標準ライブラリ・既存依存・小さな自作）を添える
- 採用時も利用箇所を限定し、将来置き換え可能な形（薄いラッパー経由等）を条件に付けられないか検討する

## やること / やらないこと

**やること**
- 導入の必要性審査（MVPへの貢献の確認）
- 既存手段（標準ライブラリ・導入済み依存・小さな自作）での代替可否の確認
- 保守性の確認（メンテナンス状況・既知の重大な問題）
- 学習コストと、将来の置き換えやすさの評価
- ライセンスの確認（プロジェクトの利用形態と両立するか）

**やらないこと**
- 実際の導入・インストール・設定作業（→ phase-implementation）
- バージョンアップ・依存更新の作業
- 導入後の実装
- 内部構造（クラス・レイヤー）の審査（→ architecture-review）

## 手順

1. **提案の確認**: 何を・何のために導入したいのか、今回のPhaseの完了条件を確認する
2. **必要性審査**: 上記2つの質問を当てる。どちらもNoなら見送りとし、手順3へ進む
3. **代替の確認**: 標準ライブラリ・導入済みの依存・小さな自作（目安: 初学者が読める数十行以内）で代替できないかを確認する。できるなら代替案を第一候補にする
4. **採用時の評価**（採用に傾いた場合のみ):
   - 保守性: メンテナンスが継続しているか・既知の重大な問題がないか
   - 学習コスト: 初学者が使い方を理解する負担がMVPに見合うか
   - 置き換えやすさ: 利用箇所を限定できるか（プロジェクト全体に染み出さないか）
   - ライセンス: プロジェクトの利用形態（個人利用・公開・商用）と両立するか
5. **判定と報告**: 出力フォーマットに従って報告する。採用時は「利用を限定する条件」を付けられる場合は明記する

## チェックリスト

- [ ] 必要性が今回のPhaseの完了条件に由来しているか（「便利そうだから」「将来使うから」になっていないか）
- [ ] 標準ライブラリ・導入済み依存で代替できないか確認したか
- [ ] 小さな自作（初学者が読める規模）で足りないか確認したか
- [ ] メンテナンス状況・既知の重大な問題を確認したか
- [ ] ライセンスがプロジェクトの利用形態と両立するか確認したか
- [ ] 学習コストがMVPの価値に見合うか
- [ ] 利用箇所を限定し、将来置き換え可能にできるか
- [ ] 同じ役割の依存が既に入っていないか（役割の重複）

### 審査内容
| 観点 | 評価 | 備考 |
|---|---|---|
| 必要性（完了条件由来か） | | |
| 既存手段での代替 | | |
| 保守性 | | |
| 学習コスト | | |
| 置き換えやすさ | | |
| ライセンス | | |

<利用箇所の限定・ラッパー経由などの条件。無ければ「なし」>

## 適用例

- 「設定ファイルの読み込みにライブラリを入れたい」の審査
- 「HTTPクライアントを自作するかライブラリを使うか」の判断
- 「テスト用のモックライブラリを追加していいか」の審査
- 外部CLIツールをビルド・実行フローに組み込む前の確認

## 適用しないケース

- 導入決定後のインストール・設定作業（→ phase-implementation）
- 既存依存のバージョンアップ作業
- クラス・ファイル分割など内部構造の審査（→ architecture-review）
- 言語・フレームワークそのものの選定（プロジェクト開始時の判断。本スキルの範囲を超える）

## 他Skillとの責務境界

- **architecture-review**: 内部構造（クラス・ファイル・レイヤー）の追加を「原則不要」から審査する。本スキルは外部依存（ライブラリ・ツール）を同じスタンスで審査する。対象が内か外かの違いで、審査の型は共通
- **phase-implementation**: 採用判定後の導入・実装を担う
- **quality-review**: 実装差分に無断で追加された依存を検知した場合、本スキルへ審査を委譲する
- **release-readiness**: 出荷前チェックの一部として、未審査の依存が混入していないかを確認する側
