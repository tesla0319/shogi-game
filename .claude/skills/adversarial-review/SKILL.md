# 敵対的レビュー スキル

## 目的

実装者の意図を好意的に補完せず、「壊すつもりで読む」レビューを行い、
再現手順つきの指摘を重大度順に報告する。レビューのみで、コード・テスト・READMEの変更は行わない。

## 手順

### 1. 差分の特定（毎回最初にやる）
- 前回レビュー時点との差分を特定する（git log / diff、無ければ関数一覧の差集合とPhaseタグコメントの新旧）
- 「新しく増えたsurface」（新コマンド・新規の外部入出力・新規の永続化・新規のスレッド）を列挙する
- レビューの重心は増分に置き、既存部分は前回指摘の消化状況を確認する

### 2. 静的に読む（優先順位つき）
以下の順で疑う。上ほど事故時の被害が大きい:
1. 外部プロセス起動・シェル解釈（subprocess / shell=True / 文字列連結）
2. パス解決（realpath / commonpath / 別ドライブValueError / トラバーサル）
3. ファイル永続化（非原子的書き込み / 壊れファイル時の挙動 / 復旧導線）
4. 例外の捕捉漏れ（requests派生 / json KeyError / import時OSError / EOFError）
5. 並行処理（スレッドからの共有資源アクセス / print割り込み / ロック有無）
6. LLMプロンプト（format対象 / デリミタ封じ込め / マーカー方式の静かな失敗 / injection耐性）
7. 長時間処理（1件の失敗で全損しないか / タイムアウト / 途中成果の保全）
8. ディスパッチ（prefix衝突 / 順序依存 / 新コマンド追加時の破綻余地）

### 3. 動的に検証する（仮説は必ず実行して確かめる)
- pytestを実行し、全件の成否と失敗理由を記録する（修正はしない）
- 「落ちるはず」と思った箇所は、モック注入で実際に想定外入力を流して再現させる
  （例: requestsの捕捉外例外、responseキー欠落JSON、壊れたファイル、オプション値内のオプション語）
- 再現できたものは「再現済み」、コードから論理的に導けるだけのものは「要実機確認」と区別して報告する
- Windows依存（clip / powershell / winsound / 別ドライブ）はLinux上で断定せず、実機確認事項として明示する

### 4. テスト戦略を評価する
- 件数ではなく「捕捉漏れ系の異常系」が空白になっていないかを見る
- 非ハーメチックなテスト（実ファイル依存 / 実行順依存 / グローバル状態リーク）を指摘する
- 「実装が直った指摘」に回帰テストが付いているかを確認する

## 報告形式

1. **総評**: 完成度 / 一番大きいリスク / 今すぐ直すべきか様子見か
2. **重大度別指摘**（Critical / High / Medium / Low）。各指摘に必ず:
   - 指摘内容と根拠箇所（行番号）
   - 具体的に起きる問題（再現済みなら再現結果を貼る）
   - 最小修正案（MVP方針に沿う数行単位のもの）
   - pytestで固定すべき回帰観点
   - 今すぐ対応すべきか
3. **やらない方がよい改善案**: 過剰設計・MVP方針と矛盾する案を明示的に却下する
4. **次に着手するならトップ3**: 理由と、Claude Codeへ渡せる小さい作業単位に分解した形

## 制約

- 調査のみ。コード・テスト・ドキュメントを変更しない
- Criticalが無いなら無いと正直に書く（指摘を盛らない）
- 修正効果を過大に書かない（例: キャッシュ化は「定常のみ短縮・初回は不変」のように正確に）
- 定数定義だけを根拠に実行時リスクを断定しない。必ず使用箇所まで追う
- 良い設計はレビュー内で明示的に認める（何を守り続けるべきかを伝えるため）
