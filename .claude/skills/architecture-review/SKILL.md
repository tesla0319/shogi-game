# architecture-review スキル

## 目的

新規クラス・ファイル・レイヤー・依存の追加や責務の変更が「今回のPhaseに必要か」を審査し、
採用可 / 縮小して採用 / 見送り の判定と、最小変更で済む代替案を返す。実装は行わない。

## 原則

- CLAUDE.md / AGENT.md の共通ルールに従う
- **新規構造は「原則不要」から出発する**。必要性が証明されたときのみ採用する（このスキルの核となるスタンス）
- 必要性の判定は次の2つの質問で行う。どちらもNoなら見送り:
  1. 追加しないと今回のPhaseの完了条件を満たせないか
  2. 追加しないと既存の責務が壊れるか（1クラス・1ファイルに無関係な責務が混ざるか）
- 見送り判定のときは、必ず最小変更の代替案を添える（判定だけで終わらない）
- 指摘の重要度は Must（放置できない）/ Should（対応推奨）/ Could（記録のみ、今は対応しない）で分類する

## やること / やらないこと

**やること**
- 構造変更（クラス・ファイル・レイヤー・依存の追加/分割/変更）の必要性審査
- 既存構造・命名・依存方向との整合確認
- 最小変更で済む代替案の提示
- 手戻りリスクの評価

**やらないこと**
- 実装・リファクタの実行（→ phase-implementation）
- 実装差分全体の品質レビュー（→ quality-review）
- テスト設計（→ test-design）
- 技術選定・フレームワーク乗り換え・全面再設計の判断

## レビュー手順

1. **変更提案の確認**: 何を追加・変更したいのか、その理由、今回のPhaseの完了条件を確認する
2. **必要性審査**: 上記2つの質問を当てる。どちらもNoなら見送りとし、手順4へ進む
3. **整合確認**: 既存の構造・命名規則・依存方向に合っているかを確認する
4. **代替案検討**: 既存クラスへの関数追加・引数追加・設定値追加で済まないかを検討する。済むなら代替案を第一候補にする
5. **手戻りリスク評価**: この構造が後のPhaseで作り直しにならないかを確認する
6. **判定と報告**: 出力フォーマットに従って報告する

## チェックリスト

- [ ] **必要性**: 追加の理由が今回のPhaseの完了条件に由来しているか（「将来使うから」になっていないか）
- [ ] **既存整合**: 既存の構造・命名規則と揃っているか
- [ ] **責務の量**: 変更後の1クラス・1ファイルの責務が増えすぎていないか
- [ ] **抽象化の妥当性**: 将来のためだけの抽象化・拡張ポイントになっていないか
- [ ] **依存方向**: 依存が一方向か。循環や、UI層からドメイン層への逆流がないか
- [ ] **代替案**: より小さい変更（関数追加・引数追加）で済む方法を検討したか
- [ ] **手戻りリスク**: 後のPhaseでこの構造を作り直す可能性はないか
- [ ] **タイミング**: 今回のPhaseでやるべき変更か。別Phase送りにできないか

## 出力フォーマット

```markdown
## 構造レビュー結果: <変更提案名>

### 判定
<採用可 / 縮小して採用 / 見送り（代替案採用）> を1行 + 理由

### 指摘一覧
| # | 重要度 | 指摘 | 理由（何が困るか） | 具体例 |
|---|---|---|---|---|

### 代替案
<最小変更で済む方法。見送り判定のときは必須>

### 手戻りリスク
<後のPhaseで作り直しになる可能性。無ければ「なし」>

### 別Phase候補
<Could相当の構造改善案。無ければ「なし」>
```

## 適用例

- 「描画処理を新しいクラスに切り出したい」の是非審査
- 「設定を辞書からクラスにすべきか」の判断
- UIとロジックのファイル分割案の審査
- quality-reviewが責務分離・過剰設計の問題を検知した後の深掘り

## 適用しないケース

- 実装差分の総合レビュー（→ quality-review）
- 関数1つの追加など、構造に影響しない小さな変更（審査不要。直接実装へ）
- フレームワーク乗り換え・アーキテクチャ全体の再設計（本スキルの範囲を超える。別途相談）
- 実装済みコードのリファクタ実行

## 既存Skillとの責務境界

- **quality-review** = Phase成果物の総合レビュー（広く浅く、完了後中心）。
  **本スキル** = 構造変更の是非のみ（狭く深く、実装前中心）
- quality-reviewの「責務分離」「過剰設計チェック」で問題が検知された場合の引き継ぎ先
  （quality-review → test-design のテスト委譲と同じ構造）
- レビューの向きが逆である点が独立理由: quality-reviewは「作った成果を評価する」、
  本スキルは「作る前の提案を疑う」
