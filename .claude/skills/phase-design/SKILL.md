# phase-design スキル

## 目的

機能要望・要件を、レビュー可能な最小単位のPhaseに分割し、
各Phaseの完了条件・依存関係・後回し候補を整理したPhase分割表として返す。実装は一切行わない。

## 原則

- CLAUDE.md の共通ルール（開発方針・コード方針・テスト方針・やらないこと）に従う
- MVP判定は次の2つの質問で行う。どちらもNoなら後回し候補とする:
  1. これが無いと動かないか？（動作必須）
  2. これが無いと価値を検証できないか？（検証必須）
- 1 Phase = phase-implementationスキルで1回で実装でき、単体で動作確認できる粒度にする
- 分割のための分割をしない。1 Phaseで完結するなら分割しない

## 手順

### 1. 要望の確認
- 実現したい価値を1〜2行で言語化し、依頼者に確認する
  （機能の列挙ではなく「誰が何をできるようになるか」）

### 2. MVP判定
- 各機能候補に上記2つの質問を当て、「動作必須 / 検証必須 / 後回し」を判定する

### 3. Phase分割
- MVPを1 Phaseの粒度に切る（目安: 触るファイルが数個、新規関数が数個以内）

### 4. 境界の明確化
- 各Phaseに「含む / 含まない」を明記する。特に「含まない」を書く（先取り実装の予防）

### 5. 依存と手戻りリスク
- Phase間の依存順と、「後のPhaseで前のPhaseを作り直す」リスクがある箇所を挙げる
- リスクが高い場合のみ、最小のインターフェース取り決め（関数シグネチャ程度）を先に提案する

### 6. 報告
- 分割表で出力し、Phase 1の完了条件について依頼者の合意を取る

## チェックリスト

- [ ] **価値の言語化**: 「誰が何をできるようになるか」を書いたか（機能名の羅列になっていないか）
- [ ] **MVP判定**: 全機能候補に「動作必須/検証必須/後回し」の判定を付けたか
- [ ] **粒度**: 各Phaseは単体で動作確認可能か。1 Phaseが大きすぎないか
- [ ] **完了条件**: 各Phaseの完了条件が1〜2行で書けているか。「〜ができる」形式か
- [ ] **含まない宣言**: 各Phaseに「このPhaseではやらないこと」を書いたか
- [ ] **依存順**: Phase間の依存が一方向か。循環していないか
- [ ] **手戻りリスク**: 後のPhaseが前のPhaseの作り直しを要求しないか。要求するなら分割を見直したか
- [ ] **過剰分割チェック**: 分割のための分割になっていないか（1 Phaseで済むものを無理に割っていないか）
- [ ] **既存スキルとの接続**: 各Phaseがそのままphase-implementationの入力になる形式か

## 出力フォーマット

```markdown
## Phase分割案: <機能名>

### 実現したい価値
<誰が何をできるようになるか。1〜2行>

### MVP判定
| 機能候補 | 判定 | 理由 |
|---|---|---|
<動作必須 / 検証必須 / 後回し>

### Phase分割表
| Phase | 完了条件 | 含む | 含まない | 依存 | 手戻りリスク |
|---|---|---|---|---|---|

### 後回し候補（今回のPhase計画に入れない）
<機能と、後回しにしてよい理由>

### 依頼者への確認事項
<Phase 1の完了条件の合意 + 未確定の仕様>
```

## 適用例

- 「棋譜の読み込み機能を追加したい。Phase分割して」
- 「AIつむぎに履歴検索を付けたいがMVPはどこまで？」
- 外部で作った要件を貼って「実装計画にして」
- 大きめの要望が来て、phase-implementationに入る前の整理

## 適用しないケース

- 分割後の実装そのもの（→ phase-implementation）
- 1 Phaseで明らかに終わる小さな修正・バグ修正（分割不要。直接phase-implementationへ）
- 技術選定・アーキテクチャ全体の再設計（Phase分割の範囲を超える。別途相談）
- 締切・工数見積もり（このスキルは順序と粒度のみ扱い、時間見積もりはしない）
