# bug-fix スキル

## 目的

不具合が「再現テストが red → 修正後 green」で証明された状態で完了すること。原因不明のまま対症療法で終わらせない。

## 原則

- 修正コードより先に、不具合を再現する失敗テストを書く
- 修正は原因に対する最小限にとどめる。周辺の気になる箇所は別途提案する
- 将棋ルール起因の不具合は、CLAUDE.md「将棋ドメインの取り決め」とルールの正（連盟規定）に照らして正しい挙動を先に確定させる
- 不明点は推測で埋めず、着手前に質問する

## 手順

### 1. 着手前（実装を書き始める前に必ず）

- 不具合の再現条件（局面SFEN・指し手・操作手順）を特定し、依頼者に確認する
- 「正しい挙動」を1〜2行で言語化する。将棋ルールが絡む場合は根拠（ルール）も添える
- 触るファイルと、触らない範囲を明示する

### 2. 実装

- まず再現テストを追加し、red になることを確認する
- 原因を特定してから修正する（原因が言えない修正はしない）
- CLAUDE.md の「実装の型」に必ず合わせる

### 3. テスト

- 再現テストが green になることを確認する
- 同種の境界値（似た局面・似た駒種）のテストを追加する
- 既存テスト全件 green を維持する

### 4. 報告（完了時に必ず含める）

- 原因の説明（何が・なぜ壊れていたか）
- 変更ファイル一覧と追加した再現テスト
- テスト結果（全件数・成否）
- 同種の不具合が潜んでいそうな箇所・残っているリスク
- CLAUDE.md「既知の制約・注意点」への追記案（ハマりどころだった場合）

## やらないこと

- 再現テストなしの修正
- 原因不明のままの「動いたからOK」修正
- 修正のついでの周辺リファクタ（別途提案する）
